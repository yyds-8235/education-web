# 智能教学平台需求规格与数据表设计

## 1. 文档说明

本文档基于当前项目代码（前端 + Redux 本地数据模型）对原始需求进行结构化整理，形成可评审、可落地的需求规格说明，并给出与现有代码类型一致的数据库表结构建议。

文档目标：
- 对齐原始需求与当前实现现状。
- 明确已实现、部分实现、待实现功能。
- 输出可用于后端建模的表结构草案。

## 2. 项目范围与角色

系统包含 3 类角色：

| 角色 | 主要模块 |
|---|---|
| 教师 | 课程系统、授课系统、测试系统 |
| 学生 | 课程系统、授课系统、测试系统 |
| 教务处（admin） | 课程表管理、考勤管理、数据统计分析、学生信息管理 |

当前基础字典（已在代码中统一）：

| 维度 | 枚举值 |
|---|---|
| 年级 | 初一、初二、初三、高一、高二、高三 |
| 学科 | 语文、数学、英语、物理、化学、多媒体、日语 |
| 课程公开范围 | public、private、class_only |

## 5. 数据库表结构设计（按现有代码模型）

设计原则：
- 主键统一 `id`（`varchar(36)`，可用 UUID/雪花 ID）。
- 时间字段统一 `created_at`、`updated_at`（`datetime`）。
- 枚举字段尽量与前端类型一一对应。
- 所有外键字段建立索引。

### 5.1 用户与身份域

#### 表：`users`

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | varchar(36) | PK | 用户 ID |
| username | varchar(64) | UK, NOT NULL | 登录账号 |

| updated_at | datetime | NOT NULL | 更新时间 |

账号规则说明：
- 所有角色统一使用一个登录接口。
- `users.username` 前缀规范：`stu*` 为学生、`tch*` 为教师、`admin` 为固定管理员账号。
- 管理员不单独建设扩展表，管理员基础信息仅保存在 `users` 表。

## 6. 关键关系与约束

1. 一个课程对应多个章节，章节对应多个资源。  
2. `course_students` 维护课程与学生多对多关系，并驱动授课签到与测试可见范围。  
3. 一个授课会话可有多个签到会话、互动记录、随机抽选记录。  
4. 一个测试可有多题目；一个学生在一个测试下建议只有一份有效提交（可覆盖更新）。  
5. 课程表调课与代课必须关联原课表项，并记录通知投递状态。  
6. 考勤原始日志与业务考勤记录建议分层存储，便于追溯与纠错。  

## 7. 与当前代码的一致性说明

1. 字段命名已尽量与 `src/types` 中模型一致。  
2. 枚举值与前端状态值保持一致，降低联调转换成本。  
3. 课程、授课、测试三大模块可优先按本表结构直接落库。  
4. 课程表、考勤、分析模块当前存在“前端演示逻辑”，后端可按本文先补齐实体与接口。
